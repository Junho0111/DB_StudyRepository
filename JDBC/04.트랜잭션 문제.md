#  트랜잭션 문제점들

### 애플리케이션 구조
* 여러 구조들중 가장 단순하며 많이 사용되는 방법은 역할에 따라 계층을 3가지고 나누는것임.

<br>

* **프레젠테이션 계층**
<br>UI와 관련된 처리
<br>웹 요청과 응답
<br>사용자 요청을 검증
<br>주된 사용 기술: 서블릿, HTTP같은 웹 기술, 스프링MVC



* **서비스 계층**
<br>비즈니스 로직
<br>주된 사용 기술: 왠만하면 특정 기술에 의존하지않고 순수 자바로 작성

* **데이터 접근 계층**
<br>실제 데이터베이스에 접근 코드
<br>주된 사용 기술: JDBC, JPA, File, Redis 등등

<br>

> @Controller(UI 관련 처리) -> @Service(비즈니스 로직) -> @Repository(DB접근기술) -> DB서버<br>각각 프레젠테이션 계층, 서비스계층, 데이터 접근 계층 으로 나뉨.

<br>

### 문제점
* 다른 계층들도 중요하지만 그중 가장 중요한 계층은 비즈니스 로직이 들어있는 서비스 계층임.
나중에 UI또는 데이터 저장 기술 등이 변경되어도 비즈니스 로직은 최대한 변경이 없어야하며 그대로 유지가 되어야하기 때문임.

* 위의 문제점을 해결하려면 서비스 계층이 특정 기술에 종속적이지 않게 개발 되어야함(오로지 비즈니스 로직 자체에 집중할 수 있도록 해야함)

> 서비스 계층이 특정 기술에 종속되지 않음 -> 비즈니스 로직 유지보수 및 테스트 수월

<br>

* 순수 JDBC를 활용한 트랙잭션 적용
```java
    public void accountTransfer(String fromId, String toId, int money) throws SQLException {
        Connection con = dataSource.getConnection();
        try {
            con.setAutoCommit(false); //트랜잭션 시작
            //비즈니스 로직
            bizLogic(con, fromId, toId, money);
            con.commit(); //성공시 커밋
        } 
        catch (Exception e) {
            con.rollback(); //실패시 롤백
            throw new IllegalStateException(e);
        } finally {
        release(con);
        }
    }

```
<br>

* 트랜잭션을 적용하며 생긴 문제들
<br>1. JDBC 구현 기술이 서비스 계층에 누수되는 문제
<br>(트랜잭션을 적용하기 위해 JDBC 구현 기술이 서비스 계층에 누수되었음)<br>
<br>2. 트랜잭션 동기화 문제
<br>(같은 트랜잭션을 유지하기 위해 커넥션을 파라미터로 넘겨야함 => 같은 기능도 트랜잭션용 기능과 트랜잭션을 유지하지 않아도 되는 기능으로 분리해야함)<br>
<br>3. 트랜잭션 적용 반복 문제

<br>

* 예외 누수
<br>1. 데이터 접근 계층의 JDBC 구현 기술 예외가 서비스 계층으로 전파된다.<br>
<br>2. SQLException 은 체크 예외이기 때문에 데이터 접근 계층을 호출한 서비스 계층에서 해당 예외를  잡아서 처리 하거나 명시적으로 throws를 통해서 다시 밖으로 던져야함.
<br>(SQLException이것도 결국 JDBC전용 기술이며 JPA와같은 기술을 쓴다면 그에 맞는 예외로 변경해야하며 서비스 코드도 수정해야함)

<br>

* JDBC반복 문제
<br>위의 코드는 순수JDBC로 작성되었으며 유사한 코드 반복이 너무 많음
<br>(try, catch, finally...커넥션 연결 PreparedStatement사용 및 sql문 매핑 실행 커넥션 정리 등)
