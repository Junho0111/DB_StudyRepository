
# JdbcTemplate 이용 - 반복 문제 해결

## 개요

* 리포지토리에서 JDBC 사용 시 발생하는 반복 코드를 제거하기 위해 `JdbcTemplate`을 도입.

<br>

## JDBC 반복 문제와 JdbcTemplate

리포지토리 메서드마다 반복되는 중복 로직을 템플릿 콜백 패턴으로 해결함.
>템플릿 콜백 패턴이란 스프링의 JdbcTemplate이 JDBC 연결, 예외 처리, 자원 반환 등 반복되는 고정 작업(템플릿)을 처리하고, 개발자는 SQL 실행 및 매핑(콜백)에 집중할  수 있게하는 디자인 패턴.

* **반복 지점**: 커넥션 조회/동기화, `PreparedStatement` 생성/바인딩, 쿼리 실행, 결과 바인딩, 예외 변환, 리소스 종료 등이 모든 메서드에서 반복됨.

<br>

* **문제점**: 위의 사항들이 계속되서 반복되는 문제가 있음.
```java
@Override
    public Member save(Member member) {
        String sql = "insert into member(member_id, money) values(?,?)";

        Connection con = null;
        PreparedStatement pstmt = null;

        try {
            con = getConnection();
            pstmt = con.prepareStatement(sql);
            pstmt.setString(1, member.getMemberId());
            pstmt.setInt(2,member.getMoney());
            pstmt.executeUpdate();
            return  member;
        } catch (SQLException e) {
//            DataAccessException ex = exTranslator.translate("save", sql, e);
//            throw ex;
            throw exTranslator.translate("save", sql, e);
        } finally {
            close(con, pstmt, null);
        }
    }

    @Override
    public Member findById(String memberId) {
        String sql = "select * from member where member_id = ?";

        Connection con = null;
        PreparedStatement pstmt = null;
        ResultSet rs = null;

        try {
            con = getConnection();
            pstmt = con.prepareStatement(sql);
            pstmt.setString(1, memberId);

            rs = pstmt.executeQuery();
            if(rs.next()){
               Member member = new Member();
               member.setMemberId(rs.getString("member_id"));
               member.setMoney(rs.getInt("money"));
               return member;
            } else {
                throw new NoSuchElementException("member not found memberId=" + memberId);
            }

        } catch (SQLException e) {
            throw exTranslator.translate("findById", sql, e);
        } finally {
            close(con, pstmt, rs);
        }
    }
```
<br>

* **JdbcTemplate 도입**: `DataSource`를 주입받아 생성하며, JDBC 개발 시 발생하는 대부분의 반복을 해결해 줌.


* **자동화 기능**: 트랜잭션을 위한 커넥션 동기화와 예외 발생 시 스프링 예외 변환기 실행을 자동으로 처리함.

* **JdbcTemplate적용**: 위와같은 반복되는 코드가 개선됨.
```java
    private final JdbcTemplate template;

    public MemberRepositoryV5(DataSource dataSource) {
        this.template = new JdbcTemplate(dataSource);
    }

    @Override
    public Member save(Member member) {
        String sql = "insert into member(member_id, money) values(?,?)";

        //반환은 영향받은 row수
        template.update(sql, member.getMemberId(), member.getMoney());
        return member;
    }

    @Override
    public Member findById(String memberId) {
        String sql = "select * from member where member_id = ?";
        return template.queryForObject(sql, memberRowMapper(), memberId);
    }

    private RowMapper<Member> memberRowMapper() {
        return (rs, rowNum) -> {
            Member member = new Member();
            member.setMemberId(rs.getString("member_id"));
            member.setMoney(rs.getInt("money"));
            return member;
        };
    }

    @Override
    public void update(String memberId, int money) {
        String sql = "update member set money = ? where member_id = ?";
        template.update(sql, money, memberId);//위와같이 업데이트된 로우수(가로)
    }
```
* rs(result) : 디비에서 조회된 row데이터
* rowNum: 현재 처리 중인 row번호(0부터 시작함)
* return: Member객체(rs에서 꺼낸 데이터를 매핑한)
* 람다식: // (입력 데이터) -> { 실행할 로직 및 반환값 }

<br>

* findById 동작과정 (템플릿적용 전 코드와 동일함)
<br>findById 호출 -> queryForObject 실행 및 데이터 뽑아옴(0행부터 실행하여 memberId와 같은값 찾음 -> 없으면 예외) -> 
<br>나온값으로 RowMapper 실행 및 Member객체 반환 -> memberId와 비교하여 반환  

<br>

## 정리: 서비스 계층의 순수성 완성

다양한 스프링 기술을 조합하여 이상적인 서비스 계층을 구축함.

| 해결 과제 | 적용 기술 | 효과 |
| --- | --- | --- |
| **트랜잭션 의존성** | 트랜잭션 추상화 + AOP | 서비스 계층에서 특정 트랜잭션 기술 의존 제거 |
| **예외 종속성** | 예외 추상화 + 예외 변환기 | 기술 변경 시에도 서비스 계층의 예외 처리 로직 유지 |
| **구현체 변경** | 리포지토리 인터페이스 의존 | DI를 통해 서비스 코드 수정 없이 구현체 교체 가능 |
| **JDBC 반복** | JdbcTemplate | 리포지토리의 중복 코드를 제거하고 핵심 로직만 남김 |


결국 dataSource에 디비주소, 이름, 비번 매핑하고 리포지토리에서 datdataSource를 받아 로직(crud 등)을 작성하고, service 계층에서 리포지토리 메서드를 호출함.
