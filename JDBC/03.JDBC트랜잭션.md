## 트랜잭션 이해

### 1. 트랜잭션

* **정의**: 트랜잭션은 '거래'라는 뜻으로, 데이터베이스 내에서 하나의 거래를 안전하게 처리하도록 보장해주는 것을 의미. 


* **동작 원리**: 여러 작업이 합쳐져서 하나의 작업처럼 동작해야 하며, 모든 작업이 함께 성공하면 **커밋(Commit)** 하고, 하나라도 실패하면 거래 전 상태로 **롤백(Rollback)** 


* **예시 (계좌이체)**: 계좌이체라는 거래는 A의 잔고 감소와 B의 잔고 증가라는 2가지 작업이 반드시 하나의 작업처럼 동작해야하며 반드시 함께 성공하거나 함께 실패해야 함 

<br>

### 2. ACID

트랜잭션은 다음의 4가지 성질을 보장해야함. 

* **원자성(Atomicity)**: 트랜잭션 내에서 실행한 작업들은 마치 하나의 작업인 것처럼 모두 성공 하거나 모두 실패해야 한다.

* **일관성(Consistency)**: 모든 트랜잭션은 일관성 있는 데이터베이스 상태를 유지해야 한다. 예를 들어 데이터베이스에서 정한 무결성 제약 조건을 항상 만족해야 한다. 

* **격리성(Isolation)**: 동시에 실행되는 트랜잭션들이 서로에게 영향을 미치지 않도록 격리한다.<br> 예를 들어 동시에 같은 데이터를 수정하지 못하도록 해야 한다. 격리성은 동시성과 관련된 성능 이슈로 인해 트랜잭션 격리 수준(Isolation level)을 선택할 수 있다. 

* **지속성(Durability)**: 트랜잭션을 성공적으로 끝내면 그 결과가 항상 기록되어야 한다.<br> 중간에 시스템에 문제가 발생해도 데이터베이스 로그 등을 사용해서 성공한 트랜잭션 내용을 복구해야 한다. 

> **참고**: 격리성을 완벽히 보장하려면 트랜잭션을 거의 순서대로 실행해야하며 이는 동시 처리 성능이 매우 나빠지므로, ANSI 표준은 이를
 4단계(READ UNCOMMITTED(커밋되지 않은 읽기), READ COMMITTED(커밋된 읽기),<br>
  REPEATABLE READ(반복 가능한 읽기), SERIALIZABLE(직렬화 가능))로 나누어 정의. 

<br>

### 3. 데이터베이스 연결 구조와 세션

* **연결 과정**: 클라이언트(was)가 서버(db) 연결을 요청하고 커넥션을 맺으면 데이터베이스 서버는 내부에 **세션(Session)** 을 생성함. 


* **역할**: 해당 커넥션을 통한 모든 SQL 요청은 이 세션을 통해 실행되며, 세션이 트랜잭션을 시작하고 종료(commit/rollback)함. 


* **커넥션 풀**: 커넥션 풀이 10개의 커넥션을 생성하면, DB 서버 내부에도 10개의 세션이 만들어짐. 

<br>

### 4. 자동 커밋 / 수동 커밋

* **자동 커밋(Auto Commit)**: 각각의 쿼리 실행 직후 자동으로 커밋을 호출(db는 기본적으로 자동커밋이 디폴드임).<br> 떄문에 여러 쿼리를 하나의 트랜잭션으로 묶을 수 없음. 


* **수동 커밋(Manual Commit)**: `set autocommit false;`를 설정하여 사용하며, 보통 이를 **"트랜잭션을 시작한다"**고 표현. <br> (커넥션 풀을 사용하는 경우에 `set autocommit true;` 를 설정하여 다시 바꿔놔야함 기본적인 커넥션은 사용후 반환하면 사라지니 상관없지만 커넥션 풀의 경우에는 반환하여 세션이 유지되어 다음 쿼리문에서 문제가 생김)


* **주의**: 수동 커밋 모드에서는 반드시 마지막에 `commit`이나 `rollback`을 직접 호출해야 데이터가 실제 반영되거나 취소됨. 

<br>

### 5. DB 락(Lock) - 개념과 실습

동시에 같은 데이터를 수정할 때 발생하는 정합성 문제를 해결하기 위해 **락(Lock)** 개념을 사용. 

* **문제**: 세션1이 트랜잭션을 시작하고 데이터를 수정하는 동안 아직 커밋을 수행하지 않았는데, 세션2에서 동시에 같은 데이터를 수정하게 되면 여러가지 문제가 발생한다. <br> 
여기서 트랜잭션의 원자성이 깨지며 또한 세션1이 중간에 롤백이라도 하는 순간 세션2는 잘못된 데이터를 수정하는 문제가 발생.

* **동작 원리**: 세션이 데이터를 수정하려고(트랜잭션 시작) 해당 로우의 락을 먼저 획득함. <br>
(먼저 요청하는 쪽이 락을 가져감)


* **대기**: 다른 세션이 이미 락을 가지고 있다면, 락을 반납(락을 가지고 있는 세션이 commit또는 rollback)할 때까지 대기. <br>
설정된 **락 타임아웃** 시간을 넘기면 예외가 발생(타임아웃시간 설정가능). <br>
락이 반납되면 최신화된 로우 값으로 수정 추가로 세션2는 트랜잭션이 진행되는 로우를 조회할 수 있음 물론 조회값은 트랜잭션이 끝나기 전이면 그전값 끝난후이면 최신화 된값.


* **조회 시 락**: 데이터베이스마다 다르지만, 보통 데이터를 조회할 때는 락을 획득하지 않고 바로 데이터를 조회할 수 있다.<br>
 예를 들어서 세션1이 락을 획득하고 데이터를 변경하고 있어도, 세션2에서 데이터를 조회는 할 수 있다.<br>( 물론 세션2에서 조회가 아니라 데이터를 변경하려면 락이 필요하기 때문에 락이 돌아올 때 까지 대기 해야함.)

 * **조회 시점 락이 필요한 경우**: 조회하는 순간의 정합성이 매우 중요하여, 트랜잭션이 끝날때까지 이 값을 읽어서 판단의 근거로 쓰이는것 조차 막아야 할때 <br>
예를 들어서 애플리케이션 로직에서 `memberA`의 금액을 조회한 다음에 이 금액 정보로 애플리케이션에서 어떤 계산을 수행한다. 그런데 이 계산이 돈과 관련된 매우 중요한 계산이어서 계산을 완료할 때 까지 `memberA`의 금액을 다른곳에서 변경하거나 근거로 쓰이면 안된다.

* **조회 시점 락 획득 방법**:  `select for update` 구문을 사용.<br>
이렇게 하면 세션1이 조회 시점에 락을 가져가버리기 때문에 다른 세션에서 해당 데이터를 변경할 수 없음.<br>
(물론 이 경우도 트랜잭션을 커밋하면 락을 반납함.)

<br>

### 6. DB 락(Lock) - 예시(조회/수정)

* **세션1**

```java
set autocommit false;
update member set money=500 where member_id = 'memberA'; //조회락x 트랜잭션(수정)
select * from member where member_id='memberA' for update; //select for update
```
* **세션2**

```java
set autocommit false;
select * from member;
update member set money=500 where member_id = 'memberA';
//조회 수정 둘다 세션1이 끝나야 동작
```

### 7. 트랜잭션 적용

애플리케이션에서 트랜잭션을 적용하려면 **비즈니스 로직**이 있는 **서비스 계층**에서 트랜잭션을 시작해야 함. 

* **서비스 계층 이유**:  비즈니스 로직이 잘못되면 해당 비즈니스 로직으로 인해 문제가 되는 부분을 함께 롤백해야 하기 때문 <br>
 트랜잭션을 시작하려면 커넥션이 필요. 결국 서비스 계층에서 커넥션을 만들고, 트랜잭션 커밋 이후에 커넥션을 종료해야 함
 
 * **주의**: 애플리케이션에서 DB 트랜잭션을 사용하려면 트랜잭션을 사용하는 동안 같은 커넥션을 유지해야함. 그래야 같은 세션을 사용할 수 있음.<br>
 커넥션을 파라미터로 전달해서 같은 커넥션이 사용되도록 유지하는것임.

<br>

* **세션유지를 위한 커넥션 파라미터 전달**

```java
//Connection con 파라미터 전달
public Member findById(Connection con, String memberId) throws SQLException {
        String sql = "select * from member where member_id = ?";

        PreparedStatement pstmt = null;
        ResultSet rs = null;

        try {
            //con = getConnection();
            pstmt = con.prepareStatement(sql);
            pstmt.setString(1, memberId);

            rs = pstmt.executeQuery();
            if(rs.next()){
                Member member = new Member();
                member.setMemberId(rs.getString("member_id"));
                member.setMoney(rs.getInt("money"));
                return member;
            } else {
                throw new NoSuchElementException("member not found memberId=" + memberId);
            }

        } catch (SQLException e) {
            log.error("db error", e);
            throw e;
        } finally {
            //close(con, pstmt, rs);
            //connection은 여기서 닫지 않는다. (위의 설명대로 서비스계층의 트랙잭션사용부분에서 세션 유지를 위해 직접 닫음)
            JdbcUtils.closeResultSet(rs);
            JdbcUtils.closeStatement(pstmt);
        }
    }

```

```java
//Connection con 파라미터 전달
public void update(Connection con, String memberId, int money) throws SQLException {
        String sql = "update member set money = ? where member_id = ?";

        //  Connection con = null;
        PreparedStatement pstmt = null;

        try {
            //con = getConnection();
            pstmt = con.prepareStatement(sql);
            pstmt.setInt(1, money);
            pstmt.setString(2, memberId);
            int resultSize = pstmt.executeUpdate();
            log.info("resultSize={}", resultSize);
        } catch (SQLException e) {
            log.error("db error", e);
            throw e;
        } finally {
            //close(con, pstmt, null);
            JdbcUtils.closeStatement(pstmt);
        }
    }

```

* **서비스계층 트랜잭션 적용**
```java
 public void accountTransfer(String fromId, String toId, int money) throws SQLException {
        Connection con = dataSource.getConnection();
        try {
            con.setAutoCommit(false); // 트랜잭션 시작
            //비즈니스 로직
            bizLogic(con, fromId, toId, money);
            con.commit();//성공시 커밋
        } catch (Exception e) {
            con.rollback();//실패시 롤백
            throw new IllegalStateException(e);
        } finally {
            release(con);
        }
    }
```
<br>

```java
private void bizLogic(Connection con, String fromId, String toId, int money) throws SQLException {
        Member fromMember = memberRepository.findById(con, fromId);
        Member toMember = memberRepository.findById(con, toId);

        memberRepository.update(con, fromId, fromMember.getMoney() - money);
        validation(toMember);
        memberRepository.update(con, toId, toMember.getMoney() + money);
    }
```
<br>

```java
private static void release(Connection con) {
        if (con !=null){
            try {
                con.setAutoCommit(true); // 커넥션 풀 고려
                con.close();//여기서 닫음(커넥션 풀의 경우 풀에 반환)
            } catch (Exception e) {
                log.info("error", e);
            }
        }
    }

```

### 8. 요약

1. **원자성 보장**: 트랜잭션 덕분에 계좌이체 중 예외가 발생해도 롤백을 통해 데이터를 안전하게 복구할 수 있음. 

2. **커넥션 유지**: 트랜잭션이 시작된 시점부터 끝날 때까지 같은 세션을 유지하기 위해 파라미터로 커넥션을 전달하는 방식이 필요. 

3. **서비스 계층 책임**: 트랜잭션은 비즈니스 로직을 결정하는 서비스 계층에서 관리하는 것이 원칙임. 

4. **문제**: 애플리케이션에서 DB 트랜잭션을 적용하려면 서비스 계층이 매우 지저분해지고, 생각보다 매우 복잡한 코드를 요구함.<br> 추가로 커넥션을 유지하도록 코드를 변경하는 것도 쉬운 일은 아님

5. **해결**: 스프링은 이런 문제 해결을 지원함.
