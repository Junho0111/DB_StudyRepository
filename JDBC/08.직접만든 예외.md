# 인터페이스 도입과 런타임 예외 전환

## 개요

* 서비스 계층의 순수성을 유지하기 위해 특정 구현 기술(JDBC 등)의 체크 예외 의존을 제거하는 방법을 다룸. 
* 인터페이스 도입 시 체크 예외가 발생하는 문제를 확인하고, 런타임 예외 전환을 통해 이를 해결함.

<br>

## 체크 예외와 인터페이스의 충돌

서비스 계층이 특정 기술에 종속되지 않으려면 예외 의존성도 해결해야 함.

* **인터페이스 오염**: 구현 클래스가 `SQLException` 같은 체크 예외를 던지려면 인터페이스 메서드에도 동일한 예외 선언이 필요함.


* **기술 종속**: 인터페이스에 `throws SQLException`이 포함되는 순간, 이는 더 이상 순수한 인터페이스가 아닌 JDBC 전용 인터페이스가 됨.

* 특정 기술에 종속되지 않은 순수한 인터페이스
```java
package hello.jdbc.repository;
import hello.jdbc.domain.Member;

public interface MemberRepositoryEx {
    Member save(Member member);
    Member findById(String memberId);
    void update(String memberId, int money);
    void delete(String memberId);
}
```
<br>

* JDBC 전용 인터페이스 (이걸 사용하는 구현체에서 예외를 던지려면 부모 즉 인터페이스에서도 예외가 선언 되어있어야함)
```java
package hello.jdbc.repository;
import hello.jdbc.domain.Member;
import java.sql.SQLException; // <== 이미 순수하지 않음 

public interface MemberRepositoryEx {
    Member save(Member member) throws SQLException;
    Member findById(String memberId) throws SQLException;
    void update(String memberId, int money) throws SQLException;
    void delete(String memberId) throws SQLException;
}
```
* **변경의 어려움**: 향후 JPA 등 다른 기술로 변경 시 인터페이스 자체를 수정해야 하는 파급 효과가 발생함.

<br>

## 런타임 예외 적용을 통한 해결

런타임 예외는 인터페이스에 별도로 선언하지 않아도 되므로 특정 기술로부터 자유로워짐.

* **MyDbException 생성**: `RuntimeException`을 상속받아 사용자 정의 언체크 예외를 만듦.

```java
package hello.jdbc.repository.ex;

public class MyDbException extends RuntimeException{

    public MyDbException() {
    }

    public MyDbException(String message) {
        super(message);
    }

    public MyDbException(String message, Throwable cause) {
        super(message, cause);
    }

    public MyDbException(Throwable cause) {
        super(cause);
    }
}

```

* **예외 변환**: 리포지토리에서 발생한 `SQLException`을 catch 하여 `MyDbException`으로 전환해 던짐.

```java
/**
 * 예외가 누수되는 문제 해결
 * 체크 예외를 런타임 예외로 변경
 * MemberRepository 인터페이스 사용
 * 구현체 및 인터페이스의 throws SQLException 제거
 */

// 기존 체크예외 처리
catch (SQLException e) {
    throw e;
    }

// 직접 만든 런타임예외로 체크예외를 감싸서 런타임으로 만들어 처리
catch (SQLException e) {
    throw new MyDbException(e);
    }
```
* 개선: 이렇게 하면 위의 리포지토리가 순수한 리포지토리로 개선됨.

* **원인 포함**: 변환 시 반드시 기존 예외를 생성자에 포함하여 스택 트레이스에서 진짜 원인을 확인할 수 있게 함 (매우 중요) 그렇지 않으면 장애가 발생하고도 로그에서 진짜 원인이 남지 않는 심각한 문제가 발생함.


* **순수한 서비스 완성**: 서비스 계층에서 `throws SQLException`이 제거되어 인터페이스에만 의존하는 순수성을 확보함.

<br>

## 데이터 접근 예외 직접 만들기

특정 DB 오류(예: 키 중복)에 대해 복구를 시도하고 싶을 때 직접 예외를 정의하여 대응함.

* **에러 코드 활용**: `SQLException`에 담긴 DB별 `errorCode`를 확인하여 오류 원인을 구분함 (H2 키 중복: 23505).
<br> **동작과정**: 데이터를 DB에 저장 혹은 다른 작업을 하다 예외/오류가 터지면 데이터베이스는 오류 코드를 반환하고, 이오류 코드를 받은 JDBC 드라이버는 SQLException을 던짐.<br>(SQLException에는 데이터베이스가 제공하는 errorCode라는 것이 들어있음) 

* **특정 예외 정의**: `MyDuplicateKeyException`을 만들어 특정 상황에서만 던지도록 함.

*기존에 사용했던 MyDbException을 상속받아서 의미있는 계층을 형성
<br>(데이터베이스 관련 예외 계층을 만들수있음)

```java
package hello.jdbc.repository.ex;

public class MyDuplicateKeyException extends MyDbException {

    public MyDuplicateKeyException() {
    }

    public MyDuplicateKeyException(String message) {
        super(message);
    }

    public MyDuplicateKeyException(String message, Throwable cause) {
        super(message, cause);
    }

    public MyDuplicateKeyException(Throwable cause) {
        super(cause);
    }
}

```

* **복구 시도**: 서비스 계층에서 특정 예외만 catch 하여 새로운 ID를 생성하는 등 예외 복구 로직을 구현함.

* 아래와 같이 특정 케이스에 대해서 예외를 잡아 처리할 수 있음

```java
//리포지토리 계층에서 특정 예외케이스에 맞게 던지면 서비스계층에서 날라온 예외를
//잡아 처리 혹은 던짐. 
catch (SQLException e) {
    //h2 db
    if (e.getErrorCode() == 23505) {
        throw new MyDuplicateKeyException(e);
        }
        throw new MyDbException(e);
        } 
```

<br>

```java
 @Slf4j
    @RequiredArgsConstructor
    static class Service {

        private final Repository repository;

        public void create(String memberId) {
            try {
                repository.save(new Member(memberId, 0));
                log.info("saveId={}", memberId);
            } catch (MyDuplicateKeyException e) {
                log.info("키 중복, 복구 시도");
                String retryId = generateNewId(memberId);
                log.info("retryId={}", retryId);
                repository.save(new Member(retryId, 0));
            } catch (MyDbException e) {
                log.info("데이터 접근 계층 예외", e);
                throw e;
            }
        }

        private String generateNewId(String memberId) {
            return memberId + new Random().nextInt(10000);
        }
    }

    //회원에는 아이디와 돈이 초기값으로 들어감
```

## 문제점
* SQL ErrorCodes는 각각의 데이터베이스마다 다름. 결과적으로 디비가 변경될때마다 에러코드를 모두 변경하여야함.
