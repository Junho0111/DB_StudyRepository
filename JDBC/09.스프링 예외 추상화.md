
# 2. 스프링의 데이터 접근 예외 추상화

## 개요

* DB마다 다른 에러 코드를 일일이 처리해야 하는 번거로움을 해결하기 위해 스프링이 제공하는 일관된 예외 계층과 자동 변환기를 이해함.

## 스프링 예외 추상화 계층

스프링은 특정 기술에 종속되지 않는 수십 가지 예외를 정리하여 제공함.

* **DataAccessException**: 스프링 데이터 접근 예외의 최상위 계층이며 런타임예외를 상속받음.


* **Transient 예외**: 일시적인 오류로, 동일한 SQL 재시도 시 성공 가능성이 있음 (예: 쿼리 타임아웃, 락).


* **NonTransient 예외**: 일시적이지 않은 오류로, 반복 실행해도 실패함 (예: SQL 문법 오류, 제약조건 위배).

* 이외에도 여러가지 있음.



## 스프링 예외 변환기 (Exception Translator)

DB 에러 코드를 적절한 스프링 예외로 자동 변환해 주는 기능을 제공함.

* **변환기 사용**: `SQLErrorCodeSQLExceptionTranslator`를 사용하여 `translate()` 메서드로 예외를 변환함.

* SQL 예외 변환기 사용 예시
```java
SQLExceptionTranslator exTranslator = new SQLErrorCodeSQLExceptionTranslator(dataSource);
DataAccessException resultEx = exTranslator.translate("select", sql, e);
```
* translate() 메서드의 첫번째 파라미터는 읽을 수 있는 설명이고,
<br> 두번째는 실행한 sql, 마지막은 발생된 SQLException을 전달하면됨.
<br> 이렇게 하면 적절한 스프링 데이터 접근 계층의 예외로 변환해서 반환해줌.

<br>

```java
@Test
    void exceptionTranslator() {
        String sql = "select bad grammer";

        try {
            Connection con = dataSource.getConnection();
            PreparedStatement stmt = con.prepareStatement(sql);
            stmt.executeQuery();
        } catch (SQLException e) {
            assertThat(e.getErrorCode()).isEqualTo(42122);

            SQLErrorCodeSQLExceptionTranslator exTranslator = new SQLErrorCodeSQLExceptionTranslator(dataSource);
            
            //org.springframework.jdbc.BadSqlGrammarException
            DataAccessException resultEx = exTranslator.translate("select", sql, e);
            
            log.info("resultEx", resultEx);
            assertThat(resultEx.getClass()).isEqualTo(BadSqlGrammarException.class);
        }
    }
```
* 이 test코드에서는 sql문법이 잘못되었으므로  BadSqlGrammarException을 반환해야함 
<br> 코드로는 DataAccessException를 반환하였지만 실제로는 BadSqlGrammarException이 반환되며 이는 assertThat을 통해 확인할 수 있음   

* **자동 판단**: 변환기는 첫 번째 인자의 설명, 두 번째 인자의 SQL, 마지막 인자의 발생 예외를 보고 적절한 예외(예: `BadSqlGrammarException`)를 반환함.

<br>

* **작동 원리**: `sql-error-codes.xml` 파일에 정의된 각 DB별 에러 코드를 대입하여 매칭되는 예외를 찾아냄.

## 추상화 적용 효과

서비스 계층은 특정 리포지토리 구현 기술이나 DB별 에러 코드에 종속되지 않게 됨.

```java
private final DataSource dataSource;

private final SQLExceptionTranslator exTranslator;

public MemberRepositoryV4_2(DataSource dataSource) {
    this.dataSource = dataSource;
    this.exTranslator = new SQLErrorCodeSQLExceptionTranslator(dataSource);
    //datasource를 넣는 이유는 안의 정보를 보고 어떤 db를 쓰는지 등을 자기가 보고 찾아 쓰기때문
}

    .
    .
    .

@Override
    public Member save(Member member) {
        String sql = "insert into member(member_id, money) values(?,?)";

        Connection con = null;
        PreparedStatement pstmt = null;

        try {
            con = getConnection();
            pstmt = con.prepareStatement(sql);
            pstmt.setString(1, member.getMemberId());
            pstmt.setInt(2,member.getMoney());
            pstmt.executeUpdate();
            return  member;
        } catch (SQLException e) {
//            DataAccessException ex = exTranslator.translate("save", sql, e);
//            throw ex; 이러한 구조로 new 해서 던지는게 아니라 바로 던질 수 있음
            throw exTranslator.translate("save", sql, e);
        } finally {
            close(con, pstmt, null);
        }
    }
```
* 이렇게 예외를 던지면 어떠한 DB로 변경되어도 한 예외에 대해서 DB별 코드를 반환받아 특정 경우의 예외를 던질 수 있음

* **유지보수성 향상**: JDBC에서 JPA로 변경되어도 스프링이 적절한 예외로 변환해 주므로 서비스 계층 코드를 유지할 수 있음.


* **일관된 복구**: 서비스 계층에서 예외를 잡아서 복구해야 하는 경우, 예외가 스프링이 제공하는 데이터 접근 예외로 변경되어서 서비스 계층에 넘어오기 때문에 필요한 경우 예외를 잡아서 복구하면됨 
<br> *한마디로 서비스 계층에서 예외 복구가 필요할 때 스프링이 제공하는 표준 예외를 잡아서 처리하면 됨.*
