# 데이터베이스 설계 (설계의 중요성)

### 1. 왜 설계가 중요한가?

데이터베이스 설계는 **"건축의 설계도"** 라고 할 수있음. 코드는 나중에 수정하기 상대적으로 쉽지만, 한 번 잘못 구축된 데이터 구조는 시간이 흐를수록 수정 비용이 기하급수적으로 늘어나기 때문.

<br>

### 2. 잘못된 설계의 3대 부작용

설계가 부실할 때 발생하는 실무적인 문제.

| 문제점 | 설명 | 결과 |
| --- | --- | --- |
| **데이터 중복** | 동일한 정보가 여러 테이블에 중복 저장됨. | 저장 공간 낭비 및 데이터 수정 시 누락 발생. |
| **데이터 불일치** | 한 곳은 수정했는데 다른 곳은 수정되지 않음. | "누가 진짜 데이터인가?"에 대한 신뢰도 붕괴. |
| **성능 저하** | 비효율적인 구조로 인해 쿼리가 복잡해지고 느려짐. | 서비스 전체의 응답 속도 저하. |

<br>

### 3. 좋은 설계의 목적

설계의 궁극적인 지향점.

1. **데이터 무결성 유지**: 데이터의 정확성과 일관성을 보장.
2. **유연한 확장성**: 비즈니스 요구사항이 변경되어도 구조 변경을 최소화할 수 있음.
3. **성능 최적화**: 효율적인 데이터 접근 경로(인덱스 등)를 고려한 구조를 만듬.

<br>

### 4. 설계의 3단계 과정

표준적인 설계 절차.

* **개념적 설계**: 사용자 요구사항을 분석하여 엔티티(Entity)와 관계(Relationship)를 도출. (ERD 작성 시작)
* **논리적 설계**: 개념적 구조를 특정 DBMS에 맞게 매핑하고 **정규화**를 통해 중복을 제거.
* **물리적 설계**: 실제 테이블을 생성하고 컬럼의 데이터 타입, 인덱스, 파티션 등을 설정하여 성능을 최적화.

<br>

### 5. 실무 포인트

* **"데이터는 한 곳에만 있어야 한다"**: 한 데이터가 여러 곳에 존재하면 반드시 언젠가 불일치 문제가 발생.
* **비즈니스 이해가 먼저**: 기술적인 정규화보다 더 중요한 것은 "서비스가 어떻게 돌아가는지" 비즈니스 로직을 완벽히 이해하는 것임.
* **데이터의 생명주기 고려**: 데이터가 생성(C)되어 수정(U)되고 삭제(D)되는 전체 흐름을 머릿속으로 그리며 설계해야함.
