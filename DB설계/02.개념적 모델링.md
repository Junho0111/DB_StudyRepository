# 데이터베이스 설계 (개념적 모델링)

### 1. 개념적 모델링이란?

현실 세계의 복잡한 비즈니스 프로세스를 추상화하여 **엔티티(Entity)**와 **관계(Relationship)** 중심으로 표현하는 단계. 특정 DBMS(MySQL, Oracle 등)를 고려하지 않고, 순수하게 "어떤 데이터가 필요하고 서로 어떻게 연결되는가"에만 집중.

<br>

### 2. 핵심 구성 요소 (3요소)

모델링의 뼈대.

| 요소 | 설명 | 비유 |
| --- | --- | --- |
| **엔티티 (Entity)** | 관리해야 할 대상 (명사형) | "누가/무엇을?" (회원, 상품, 게시글) |
| **속성 (Attribute)** | 엔티티가 가지는 세부 정보 | "어떤 정보를?" (이름, 가격, 가입일) |
| **관계 (Relationship)** | 엔티티 간의 연관성 (동사형) | "어떻게 연결되는가?" (주문하다, 소속되다) |

<br>

### 3. 엔티티 도출 방법

요구사항 명세서에서 **'핵심 명사'**를 찾는 것부터 시작하여야함.

* **예시 (쇼핑몰)**: "고객이 상품을 주문한다."
* **고객**(엔티티) - **주문**(관계) - **상품**(엔티티)


* **주의사항**: 너무 세세한 정보(전화번호, 주소 등)는 엔티티가 아니라 속성으로 분류해야함.

<br>

### 4. 관계 정의와 차수 (Cardinality)

엔티티 간에 수량이 어떻게 대응되는지를 정의함.

1. **1:1 (일대일)**: 양쪽이 서로 하나씩만 연결됨. (예: 사용자 - 프로필)
2. **1:N (일대다)**: 한쪽은 하나, 반대쪽은 여러 개 연결됨. (예: 부서 - 사원, 고객 - 주문)
3. **N:M (다대다)**: 양쪽 모두 여러 개와 연결될 수 있음. (예: 학생 - 과목)

<br>

### 5. 실무 포인트

* **추상화의 힘**: 처음부터 테이블 컬럼 타입(INT, VARCHAR 등)을 고민하지 않아도됨. 큰 그림(엔티티와 관계)을 먼저 잘 그려야 나중에 데이터가 꼬이지 않음.
* **다대다(N:M)의 발견**: 개념적 모델링 단계에서는 다대다 관계가 자연스럽게 나타남. 하지만 실무(논리적 설계)로 넘어갈 때는 이를 반드시 1:N 관계로 풀어내야 한다는 점을 미리 염두에 두어야함.
* **커뮤니케이션 도구**: ERD는 개발자 혼자 보는 것이 아니라 기획자, 클라이언트와 데이터 구조를 논의하는 도구임. 누구나 이해할 수 있게 명확한 이름을 붙이는 것이 중요함.
